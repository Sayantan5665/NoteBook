# 08 — Extension Points

---

## 1. Purpose

This document outlines the extensibility of the Embeddings & Retrieval module. It defines the conceptual integration surfaces where future capabilities, alternative providers, and advanced retrieval paradigms can be introduced without violating existing architectural contracts.

## 2. Provider Extensions

### 2.1 Multiple Embedding Providers
The architecture is designed to support multiple, interchangeable Embedding Providers. The module abstracts over provider specifics through a conceptual provider interface.

### 2.2 Local Embedding Models
- A local provider operates entirely on-device, requiring no network connectivity.
- This extension supports offline-first use cases and preserves user privacy by ensuring no content leaves the device during embedding generation.

### 2.3 Cloud Embedding Providers
- A cloud provider dispatches content to an external API and receives high-quality semantic vectors in return.
- This extension supports users who prefer performance and model quality over strict local processing.

### 2.4 Embedding Versioning
- As embedding models evolve, stored vectors generated by an older model become incompatible with vectors generated by a newer one.
- The architecture supports an Embedding Version concept — a label attached to each stored vector indicating which model generation produced it.
- When a model is upgraded, the system can migrate by scheduling a full or incremental rebuild, producing fresh vectors under the new version label without discarding the old ones until migration is complete.

## 3. Retrieval Extensions

### 3.1 Hybrid Retrieval
- Combines semantic vector similarity from the Embedding Store with keyword relevance from the Search module.
- Results from both systems are merged and re-ranked into a unified candidate list, producing results that satisfy both conceptual proximity and exact term matching.

### 3.2 Multi-Stage Retrieval
- A two-pass retrieval strategy: a broad, fast first pass identifies a large candidate pool (e.g., top 100), followed by a deep, slower re-ranking pass over that reduced set.
- This improves result quality for complex or ambiguous queries without sacrificing overall system performance.

### 3.3 Cross-Workspace Retrieval
- Extends the retrieval scope beyond the active Workspace to search across multiple independent Workspaces.
- This extension must respect Workspace isolation and access controls defined by the Workspace module.

### 3.4 Knowledge Graph Integration
- Combines semantic proximity with structural relationship signals from the Wiki Links module.
- Notes that are semantically similar AND structurally connected (linked by Wiki Links) may receive a compound relevance boost.

### 3.5 Plugin Retrieval Providers
- Third-party plugins may register their own retrieval sources (e.g., external knowledge bases, company wikis), injecting additional candidates into the Retrieval Pipeline alongside native Notebook results.

### 3.6 Future Multi-Modal Retrieval
- Extends embedding generation beyond text to include image and audio content (where OCR or transcription pipelines are insufficient).
- A future image embedding provider would allow retrieval of visually similar Attachments.

### 3.7 Future AI-Assisted Retrieval
- An AI module may augment retrieval by pre-processing the query (e.g., query expansion) or post-processing the result set (e.g., re-ranking based on user intent inference).
- In all cases, the AI module remains a consumer of retrieval outputs — it does not own or replace the Retrieval Pipeline.

## 4. Business Rules

- **Augmentation, Not Replacement:** All extension points augment existing embedding and retrieval capabilities. They NEVER replace the foundational semantic retrieval or break existing consumer contracts.
- **Backward Compatibility:** Every extension must be introduced as an additive capability. The fundamental pipeline — Request → Embedding Comparison → Ranked Candidates → Result Set — must remain intact.
- **Ownership Unchanged:** Regardless of the extension (e.g., a plugin provider or a cloud model), ownership boundaries remain unchanged. Extended capabilities consume Notebook data on a read-only basis and NEVER assert ownership over Notes, Attachments, or OCR Results.
- **Provider Isolation:** Introducing a new Embedding Provider must not require changes to the Retrieval Pipeline or the Context Assembly stages.

## 5. Acceptance Criteria

- Switching from a Local Embedding Provider to a Cloud Embedding Provider triggers a full embedding rebuild but does not alter the Retrieval Pipeline, Context Assembly, or any canonical Notebook data.
- A Plugin Retrieval Provider can inject additional candidates into the result set without requiring modifications to the core semantic retrieval logic or the embedding store schema.
- Implementing Multi-Stage Retrieval improves result ranking but the Result Set structure returned to consumers remains identical, preserving full backward compatibility.
