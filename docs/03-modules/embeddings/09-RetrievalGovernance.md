# 09 — Retrieval Governance

---

## 1. Purpose

This document establishes the governance principles for the Embeddings & Retrieval module. It enforces ownership boundaries, defines architectural constraints, and guarantees that this module never becomes a point of canonical data mutation or AI response generation.

## 2. Module Ownership

### 2.1 Owned Entities
The Embeddings & Retrieval module strictly owns:
- **Embedding Concepts:** The theoretical definition of what an embedding is and how it relates to canonical content.
- **Embedding Lifecycle:** The orchestration of Requests, Jobs, Vectors, Refreshes, Regenerations, and Deletions.
- **Retrieval Concepts:** The theoretical definition of what retrieval is and how it differs from Search and AI generation.
- **Retrieval Requests:** The immutable, ephemeral objects representing a consumer's retrieval intent.
- **Retrieval Results:** The transient ranked candidate sets returned to consumers.
- **Context Assembly:** The process of composing a derived Context Package from Retrieval Results.

### 2.2 Unowned Entities
The Embeddings & Retrieval module strictly does **NOT** own:
- Notes
- Attachments
- OCR Results
- Tags
- Wiki Links
- Search Indexes
- AI Conversations
- LLM Interactions
- Synchronization State
- User-Authored Content of any kind

## 3. Architectural Constraints

### 3.1 Embeddings Remain Derived Artifacts
- Embeddings are numeric representations derived from canonical content. They never become part of the user's Notebook corpus.
- The embedding store is entirely disposable. A complete deletion and rebuild must be achievable at any time without data loss.

### 3.2 Retrieval Prepares Context Only
- The Retrieval Pipeline surfaces relevant entity references. It NEVER generates, synthesizes, or authors new content.
- AI response generation is the exclusive responsibility of the AI module. This module provides the raw material; it does not craft the response.

### 3.3 AI Consumes This Module
- The AI module is a downstream consumer of this module's outputs (Retrieval Results, Context Packages).
- This module has no dependency on the AI module. The relationship is strictly unidirectional.

### 3.4 Read-Only Posture
- At every stage — embedding generation, retrieval execution, and context assembly — this module operates on a strictly read-only basis with respect to canonical modules (Notes, Attachments, OCR, Tags, Wiki Links).
- The module has no architectural authority to modify, delete, or overwrite canonical data.

### 3.5 No User-Visible Authorship
- Content appearing in a Context Package is always sourced from user-authored, canonical Notes or their derivatives (OCR text, Tag names). It is never generated by this module.
- The module must preserve attribution: every Context Fragment retains a reference to its source entity UUID.

## 4. Module Relationships

| Module | Relationship | Direction |
|---|---|---|
| Notes | Consumes text payloads for embedding | Read-only inbound |
| Attachments | Consumes metadata | Read-only inbound |
| OCR | Consumes extracted text for embedding | Read-only inbound |
| Tags | Consumes Tag names as semantic signals | Read-only inbound |
| Wiki Links | Consumes link structures as ranking signals | Read-only inbound |
| Search | Optionally consumes Search Results as pre-filter | Read-only inbound |
| AI Module | Supplies Retrieval Results and Context Packages | Outbound only |

## 5. Versioning and Evolution Philosophy

### 5.1 Versioning Philosophy
- As the module evolves (e.g., migrating from one embedding model to another, or introducing multi-stage retrieval), the external contracts with consumers (AI, UI) and producers (Notes, OCR) must remain stable.
- Consumers must receive the same conceptual structure — a ranked list of entity UUID references — regardless of the underlying retrieval mechanism.

### 5.2 Backward Compatibility
- Future extensions (e.g., Multi-Modal Embeddings, Cross-Workspace Retrieval) must be introduced as additive capabilities.
- Existing consumers must not require code changes to continue functioning after a new extension is introduced.

### 5.3 Evolution Guidelines
- New Embedding Providers may be added without altering the Retrieval Pipeline.
- New Retrieval Pipeline stages may be added without altering the Context Assembly process.
- New Context Sources may be added without altering the Retrieval Pipeline.
- Governance boundaries — owned vs unowned entities — may not be expanded without a formal architectural review.

## 6. Failure Philosophy

Failures in the Embeddings & Retrieval module affect ONLY retrieval and context assembly quality.

Failures in this module MUST NEVER corrupt or alter:
- Notes
- Attachments
- OCR Results
- Tags
- Wiki Links
- Search Indexes
- Editor content

If the embedding store becomes unavailable or retrieval fails, the user retains full access to their Notebook. Discovery capabilities degrade gracefully; canonical authoring capabilities remain completely unaffected.

## 7. Business Rules

- **Absolute Data Safety:** Because embeddings are derived, the ultimate fallback for any unrecoverable error is to delete the embedding store and rebuild it from canonical sources.
- **Boundary Preservation:** Governance boundaries must remain stable regardless of technological evolution. New AI models may improve retrieval quality, but they do not shift data ownership.
- **Consumer Independence:** Consumers of retrieval outputs interpret and use results independently. This module does not prescribe how results are presented or used.

## 8. Acceptance Criteria

- A catastrophic failure of the embedding store (e.g., database corruption) renders retrieval and Related Notes unavailable, but the user can continue opening, editing, and saving Notes without interruption.
- An architectural audit confirms that the Embeddings & Retrieval module possesses zero write-access to the database tables owned by the Notes, Attachments, OCR, Tags, and Wiki Links modules.
- Introducing a new Cloud Embedding Provider does not alter the structure of the Retrieval Result Set delivered to the AI module, confirming consumer backward compatibility.
