# 01 — OCR Overview

---

## 1. Purpose

This document establishes the conceptual boundaries of the OCR module. It defines how the system interacts with binary files to extract text while strictly preserving data ownership.

## 2. Concept

OCR is an asynchronous processing service. It reads binary attachments that contain visual text (like a scanned receipt) and outputs a machine-readable text string.

## 3. Ownership

- The **OCR module** owns the processing job queue and the lifecycle of the extracted text results.
- The **Attachments module** owns the binary files being read.
- **Rule:** OCR consumes Attachments. OCR never owns Attachments. OCR never modifies binary files.

## 4. OCR Identity Philosophy
It is critical to distinguish the conceptual identities within the OCR domain:
- **OCR Request:** Represents a request to process a specific Attachment.
- **OCR Job:** Represents the active execution of that request.
- **OCR Result:** Represents the derived textual output generated by the Job.
- **Distinction:** Each concept has a distinct responsibility and lifecycle. The identity of the underlying Attachment remains entirely independent from the identity of the OCR Request, Job, or Result.

## 5. OCR Processing Philosophy
- OCR derives machine-readable textual representations from supported Attachments.
- OCR processing depends entirely on the source Attachment and the specific OCR configuration at the time of execution.
- OCR quality may improve as OCR technology or configuration evolves.
- Because of this, regenerating OCR Results is considered a normal lifecycle event, rather than an error state.

## 6. Supported Attachment Concepts

The module conceptually processes:
- Image files (e.g., JPEG, PNG, TIFF)
- Document files (e.g., PDF) containing rasterized images.

It intentionally ignores files lacking visual text properties (e.g., Audio files, ZIP archives).

## 7. Core Workflow

### 5.1 OCR Requests
- A request is dispatched when a supported Attachment is ingested, or when a user manually triggers processing. The request contains the Target Attachment UUID.

### 5.2 OCR Processing
- The module safely reads the binary stream from the storage layer using the Attachment UUID.
- It performs mathematical/algorithmic extraction without altering the source bytes.

### 5.3 OCR Completion
- Upon success, the module yields a derived text artifact and publishes an event.

### 5.4 OCR Failures
- If extraction fails (e.g., the image is too blurry, or the engine crashes), the failure is recorded. The failure has zero impact on the validity or availability of the underlying Attachment.

### 5.5 OCR Retries
- Failed jobs can be retried automatically (e.g., after a system restart) or manually by the user.

## 8. Business Rules

- **Read-Only:** The module possesses strictly read-only access to the Attachment binary.
- **Stateless Engines:** The underlying OCR processing engine should be conceptually stateless; given the same input image and configuration, it should produce the same output.
- **Background Processing:** OCR must never block the main thread or prevent a Note from being saved.

## 9. Acceptance Criteria

- When an image is passed to the OCR module, the original file hash of the image remains identical before, during, and after processing.
- A crashing OCR thread safely terminates the specific job but does not crash the Note Editor or the Attachments module.
