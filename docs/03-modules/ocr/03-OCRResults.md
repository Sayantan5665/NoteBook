# 03 — OCR Results

---

## 1. Purpose

This document defines the output generated by the OCR processing pipeline. It emphasizes that the results are entirely derivative and exist purely to enrich downstream modules like Search.

## 2. Result Concepts

### 2.1 Extracted Text
- The primary output is a raw string of text interpreted from the binary image.

### 2.2 Confidence Concepts
- The module conceptually tracks a confidence score (e.g., 0% to 100%) indicating the engine's certainty in its extraction accuracy.
- Low-confidence results might be flagged to prevent polluting the Search index with garbage characters.

### 2.3 Language Detection (Future)
- The module will detect and append the primary language of the text (e.g., "en", "fr") to assist in localized search tokenization.

### 2.4 Structured Output (Future)
- Future engines may output bounding boxes, layout coordinates, or HOCR formats, allowing the UI to overlay selectable text directly on top of the image.

## 3. Result Ownership and Lifecycle

- **Result Ownership:** The OCR module owns the result artifact.

### 3.1 OCR Result Lifecycle
OCR Results possess a conceptual lifecycle that evolves independently from the Attachment:

`Requested` &rarr; `Processing` &rarr; `Completed` &rarr; `Superseded` &rarr; `Archived` &rarr; `Regenerated`

- **Derived Nature:** OCR Results are derived artifacts. They evolve independently as processing pipelines improve.
- **Never Canonical:** OCR Results NEVER become canonical Attachment data.

### 3.2 Attachment Independence
- **Canonical Source:** Attachments remain the canonical source of truth at all times.
- **No Reverse Deletion:** Deleting OCR Results (e.g., clearing the cache) NEVER deletes Attachments.
- **Invalidation:** Removing or deleting an Attachment conceptually invalidates and destroys associated OCR Results.
- **Ownership Boundaries:** OCR Results NEVER own Attachments. The relationship is strictly unidirectional (OCR Result describes Attachment).

## 4. Business Rules

- **No Note Mutation:** OCR results are NOT appended to the Note's Markdown payload. They remain in the OCR/Metadata layer to prevent mutating user-authored text.
- **Transparency:** The fact that text was generated by a machine (and may contain errors) should be clear to downstream consumers.

## 5. Acceptance Criteria

- The OCR process yields a text string that is stored in an independent table/index tied to the Attachment UUID, leaving the user's Note completely untouched.
- Completely deleting the OCR results database table does not result in the loss of any canonical user data or attachments; the module simply begins queuing them for regeneration.
