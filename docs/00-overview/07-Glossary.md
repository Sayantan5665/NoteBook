## 1. Purpose

This document provides authoritative definitions for all domain-specific terms, acronyms, and product-specific concepts used across the Notebook documentation. All documents in `docs/` **shall** use terms consistently with the definitions provided here.

When a term is used in another document, it is assumed to carry the meaning defined here unless that document explicitly provides a narrowing definition with a local scope annotation.

---

## 2. RFC 2119 Keyword Definitions

All documents in this project use the following keywords in accordance with [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119):

| Keyword | Meaning |
|---|---|
| **SHALL** | An absolute requirement of the specification. |
| **SHALL NOT** | An absolute prohibition of the specification. |
| **SHOULD** | A strong recommendation. There may be valid reasons to deviate, but the implications must be understood and carefully weighed. |
| **SHOULD NOT** | A strong recommendation against. Deviation may exist in valid circumstances. |
| **MAY** | Indicates an optional capability that is permissible but not required. |

## 3. Core Product Concepts

### AI Chat
A conversational interface within a Workspace that allows users to ask natural language questions. Responses are generated by the local AI subsystem and are grounded exclusively in the user's own Workspace content via RAG. See [04-FunctionalRequirements.md §10](./04-FunctionalRequirements.md).

### Attachment
A file associated with a Note. Attachments are stored in the local Workspace directory. Supported types include images, PDFs, Word documents, spreadsheets, and plain text. See [04-FunctionalRequirements.md §6](./04-FunctionalRequirements.md).

### Automatic Backlink
A bidirectional link automatically maintained by the system. When Note A contains a Wiki Link to Note B, a backlink from Note B to Note A is created and updated automatically by the application without user intervention. Backlinks improve note navigation, surface related content, and are exposed to the AI retrieval subsystem as context signals to improve RAG response relevance. See [04-FunctionalRequirements.md §5](./04-FunctionalRequirements.md).

### Backlink
See *Automatic Backlink*.

### Embedding
A fixed-size numerical vector representation of a piece of text content (a note or attachment), generated by a local embedding model via Ollama. Embeddings capture semantic meaning and are used for Semantic Search and RAG retrieval. See [04-FunctionalRequirements.md §9](./04-FunctionalRequirements.md).

### Folder
A named container within a Workspace that organizes Notes. Folders may be nested to arbitrary depth. See [04-FunctionalRequirements.md §4](./04-FunctionalRequirements.md).

### Full-Text Search (FTS)
A search method that matches query terms against an index of note titles, note body text, and OCR-extracted attachment text. Implemented using SQLite FTS5. See [04-FunctionalRequirements.md §8](./04-FunctionalRequirements.md).

### Local-First
A design principle in which all user data is stored and processed on the user's local machine by default. Network connectivity is not required for core operations. Distinct from *Offline First* in that local-first additionally asserts data ownership — the local copy is the source of truth. See [01-Vision.md §4.2](./01-Vision.md).

### Note
The primary unit of user-created content in Notebook. A Note has a title, a rich text body, optional attachments, optional tags, and belongs to a Workspace (and optionally to a Folder). See [04-FunctionalRequirements.md §5](./04-FunctionalRequirements.md).

### Offline-First
A design principle in which every core feature of the application is fully operational without an active internet connection. The application degrades gracefully when connectivity is unavailable. See [01-Vision.md §4.1](./01-Vision.md).

### OCR (Optical Character Recognition)
The automated extraction of text from image-based content (images, scanned PDFs). Performed locally using Tesseract OCR. Extracted text is indexed for Full-Text Search and Semantic Search. See [04-FunctionalRequirements.md §7](./04-FunctionalRequirements.md).

### Ollama
An open-source, locally hosted AI inference runtime that enables running large language models (LLMs) and embedding models on the user's machine without cloud connectivity. Notebook uses Ollama as the default AI provider for both chat and embedding generation.

### Plugin
A discrete software extension that integrates with the Notebook Plugin System to add or replace application functionality. Plugins declare required permissions and are installed from the local filesystem. See [04-FunctionalRequirements.md §18](./04-FunctionalRequirements.md).

### Plugin System
The infrastructure within Notebook that enables plugins to be installed, managed, enabled, and disabled. The Plugin System defines the extension points and the security boundary for plugin execution. The Plugin SDK is specified in `docs/sdk/`.

### Privacy-First
A design principle asserting that user data shall never leave the local machine without explicit, per-action user consent, and that the application shall not collect or transmit any telemetry or behavioral data by default. See [01-Vision.md §4.3](./01-Vision.md).

### RAG (Retrieval-Augmented Generation)
An AI pattern in which relevant documents are retrieved from a local knowledge base (using Semantic Search) and provided as context to a language model to generate a grounded, source-attributed response. Notebook's AI Chat is exclusively RAG-based. See [04-FunctionalRequirements.md §10](./04-FunctionalRequirements.md).

### Rich Text Editor
The in-application editor for Note body content. Built on Tiptap. Supports headings, lists, tables, code blocks, formatting, embedded images, and wiki-style internal links. See [04-FunctionalRequirements.md §5](./04-FunctionalRequirements.md).

### Second Brain
A colloquial term, popularized in productivity literature, for a personal external knowledge management system that supplements human memory. Used in product descriptions and marketing to convey the purpose of Notebook. Not a technical term.

### Semantic Search
A search method that uses vector similarity between an embedding of the user's query and pre-computed embeddings of notes and attachments to return conceptually related content, regardless of exact keyword overlap. See [04-FunctionalRequirements.md §9](./04-FunctionalRequirements.md).

### Sync / Synchronization
The optional, user-initiated process of copying Workspace data to and from a user-owned Google Drive account. Google Drive is used exclusively as a synchronization target and is not the primary data store. The local SQLite database and filesystem are always the authoritative data store. Sync is bidirectional and requires explicit user authorization. It is never automatic or background by default. See [04-FunctionalRequirements.md §14](./04-FunctionalRequirements.md).

### Tag
A user-defined label applied to a Note or Attachment to categorize or annotate it. Tags are Workspace-scoped strings. See [04-FunctionalRequirements.md §13](./04-FunctionalRequirements.md).

### Tiptap
An open-source, headless, extensible rich text editor framework built on ProseMirror, used as the foundation for Notebook's Note editor.

### Todo
A task item within a Workspace. Each Todo has a title and a completion status, and may have a due date, priority, and association with a specific Note. See [04-FunctionalRequirements.md §12](./04-FunctionalRequirements.md).

### Trash
A per-Workspace holding area for soft-deleted Notes, Folders, and Attachments. Items in Trash are recoverable until permanently deleted. See [04-FunctionalRequirements.md §17](./04-FunctionalRequirements.md).

### Version History
An automatic, immutable log of saved states for each Note within a Workspace. Users may view, compare, and restore any prior version. See [04-FunctionalRequirements.md §11](./04-FunctionalRequirements.md).

### Wiki Link
An internal hyperlink between Notes within the same Workspace, written in the format `[[Note Title]]`. Resolving a Wiki Link navigates to the referenced Note. Wiki Links automatically generate *Automatic Backlinks*, improving navigation, related-note discovery, and AI context retrieval. See [04-FunctionalRequirements.md §5](./04-FunctionalRequirements.md).

### Workspace
The primary organizational and data isolation unit in Notebook. Notebook is **Workspace-first**: the Workspace is the top-level logical container and every user data entity belongs to exactly one Workspace. A Workspace is a self-contained collection of Folders, Notes, Attachments, AI Chats, Embeddings, Todos, Tags, Settings, and Version History. Each Workspace is stored as a local directory containing an embedded SQLite database. Multiple Workspaces may exist on the same machine. Each Workspace supports the following lifecycle operations independently: create, rename, delete, export, import, backup, restore, and synchronize to Google Drive. See [04-FunctionalRequirements.md §3](./04-FunctionalRequirements.md).

### Workspace-First
An architectural design principle in which the Workspace is the top-level organizational and data isolation unit. All features, data entities, settings, and AI operations are scoped to a Workspace. Each Workspace is fully independent: its creation, deletion, export, backup, and synchronization do not affect any other Workspace. See [03-Scope.md §3.1](./03-Scope.md).

## 4. Technology Terms

| Term | Definition |
|---|---|
| **Angular** | The TypeScript-based frontend framework used to build the Notebook UI. |
| **Electron** | A framework for building cross-platform desktop applications using web technologies (Chromium + Node.js). |
| **FTS5** | The full-text search extension for SQLite used to implement Notebook's keyword search. |
| **Prisma** | The ORM (Object-Relational Mapper) used to manage the SQLite database schema, migrations, and queries. |
| **SQLite** | A self-contained, serverless, embedded relational database engine. Used as Notebook's primary data store. |
| **sqlite-vec** | A SQLite extension that adds vector storage and nearest-neighbor search capabilities. Used for Semantic Search. |
| **Tesseract OCR** | An open-source OCR engine used to extract text from images and scanned documents locally. |
| **TypeScript** | A statically typed superset of JavaScript used throughout the Notebook codebase. |
| **WAL (Write-Ahead Logging)** | A SQLite journal mode that improves crash safety and concurrent read performance. Enabled for all Workspace databases. |

## 6. Out-of-Scope Terms

The following terms are explicitly **not** applicable to Notebook and **shall not** be used in documentation without a clear qualifier indicating they are out of scope:

| Term | Why Excluded |
|---|---|
| Backend / Server | No developer-owned server exists. |
| Cloud database | All data is local. |
| Authentication server | No centralized identity management. |
| Telemetry | Not collected by default; prohibited. |
| SaaS | Notebook is not a Software-as-a-Service product. |
| Subscription | No subscription model; all core features are free. |
| Graph view / Canvas | Explicitly excluded from scope. See [03-Scope.md §5](./03-Scope.md). |
| Templates | Explicitly excluded from scope. See [03-Scope.md §5](./03-Scope.md). |
