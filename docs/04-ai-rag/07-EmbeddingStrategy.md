# 07 — Embedding Strategy

> **Module:** AI & RAG
> **Status:** Frozen
> **Version:** 1.0
> **Architecture Review:** Approved
> **Applies To:** Notebook Application

---

## 1. Purpose

The Embedding Strategy expands upon the conceptual approach for generating and managing the vector representations used by the Search and Retrieval modules.

---

## 2. Conceptual Strategy

- **Generation:** When canonical Notebook entities (Notes) are created or updated, their text is chunked and passed to an embedding model to generate high-dimensional vectors.
- **Regeneration:** Because embeddings are derived artifacts, they can be safely regenerated at any time. If the vector database is corrupted, or the user switches to a new embedding model, the system simply re-embeds the canonical notes.
- **Versions:** Embedding models evolve. The system must track which model version was used to generate a specific vector. Vectors generated by different models cannot be compared via cosine similarity.

---

## 3. Consistency

- **Canonical Dominance:** Notebook entities remain the canonical source of truth at all times. If a Note's content diverges from its embedding (e.g., due to an offline sync conflict), the embedding is discarded and regenerated. The Note is never altered to match the vector store.

---

## 4. Future Embedding Providers

The generation of embeddings follows the same abstraction philosophy as the LLMs. Future embedding providers (local or remote) can be plugged in. The system must accommodate background batch jobs to re-index the workspace when a provider changes.

---

## 5. Business Rules

- **Embeddings are ephemeral derived artifacts.** They never become Notebook entities.
- **Vectors from different model versions are strictly isolated.**

---

## 6. Acceptance Criteria

- A user can completely delete the vector database file, and the application will gracefully fall back to lexical search while rebuilding the embeddings in the background from the canonical Notes.
